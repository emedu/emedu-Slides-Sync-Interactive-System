# 執行計畫 - 伊美系統 v10.0 重構

## 0. 目標說明
將「伊美全自動互動學習追蹤系統」從單一檔案的 v9.4.2 代碼庫重構為技術開發白皮書中描述的模組化 MVC 架構 (v10.0)。新系統將支援透過簡報中的 QR Code 進行即時互動，適用於直播與預錄課程場景。

## 1. 使用者審查事項
> [!IMPORTANT]
> 此重構將顯著改變專案結構。原始的單一腳本將被拆解為多個 `.gs` 和 `.html` 檔案。

> [!NOTE]
> 本計畫著重於在本地生成檔案結構與程式碼邏輯。部署至 Google Apps Script 需手動複製這些檔案至 GAS 編輯器。

## 2. 預計變更

### **後端結構 (.gs)**

後端將拆分為專門的服務模組，以提升維護性與擴展性。

#### [NEW] `Main.gs`
- **角色**: 入口點與控制器 (Controller)。
- **職責**:
    - `doGet(e)`: 處理 Web App 請求 (QR Code 掃描)。
    - 根據查詢參數將請求路由至適當的視圖。
    - `doPost(e)`: 處理表單提交 (若未來從標準 Google 表單遷移至自訂 HTML 表單)。

#### [NEW] `Service_DB.gs`
- **角色**: 資料庫抽象層。
- **職責**:
    - 封裝所有 `SpreadsheetApp` 互動。
    - 管理「主控台試算表」與「活動資料簿」。
    - 功能函式: `getStudentData` (取得學員資料), `saveResponse` (儲存回應), `getActivityConfig` (取得活動設定)。
    - **遷移**: 整合 v9.4.2 的 `install_ControlSheet` 與屬性管理邏輯。

#### [NEW] `Service_Engine.gs`
- **角色**: 業務邏輯核心。
- **職責**:
    - 處理評分邏輯 (從 v9.4.2 `Scoring` 類別遷移)。
    - 管理階段轉換與進度追蹤。
    - **遷移**: 整合 `onFormSubmit_RunV94` 與 `Scoring` 物件的邏輯。

#### [NEW] `Service_Security.gs`
- **角色**: 安全與驗證。
- **職責**:
    - 頻率限制 (防秒刷 `SUBMIT_GUARD_SECONDS`)。
    - 身份驗證 (學號檢查)。
    - **遷移**: 提取 `withSubmitGuard_` 與 `findMasterIdByFormId` 邏輯。

#### [NEW] `Service_AI.gs`
- **角色**: AI 整合。
- **職責**:
    - 與 Gemini API 介接 (目前為 placeholder/stub)。
    - 生成回饋與摘要的功能函式。

### **前端結構 (.html)**

#### [NEW] `UI_Portal.html`
- **角色**: 學員主視圖。
- **特性**: 響應式行動優先設計，顯示當前任務/問題。

#### [NEW] `UI_Action.html`
- **角色**: 用戶端邏輯。
- **特性**: 表單驗證、非同步提交至 `google.script.run` 的 JavaScript。

#### [NEW] `UI_Style.html`
- **角色**: 共用樣式。
- **特性**: 定義伊美品牌色 (海軍藍 #002147)、CSS 變數。

## 3. 驗證計畫

### 手動驗證
- **程式碼審查**: 確保所有 v9.4.2 的業務規則（評分模式：完全比對、部分給分等）皆已遷移至 `Service_Engine.gs`。
- **結構檢查**: 確認 v10.0 白皮書中指定的所有檔案皆已建立。
- **邏輯演練**: 追蹤假設的使用者流程:
    1.  使用者掃碼觸發 `Main.gs` `doGet`。
    2.  `Main.gs` 呼叫 `Service_DB` 檢查活動狀態。
    3.  `Main.gs` 回傳 `UI_Portal` 頁面。
    4.  使用者提交答案 -> `UI_Action` 呼叫 `Service_Engine`。
    5.  `Service_Engine` 進行評分並透過 `Service_DB` 儲存結果。
