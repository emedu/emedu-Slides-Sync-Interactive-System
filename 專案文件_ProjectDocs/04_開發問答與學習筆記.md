# 04\_開發問答與學習筆記 — emedu-Slides-Sync-Interactive-System

> **最後更新**：2026-03-04 (v10.1.5 FINAL FIX)
> **格式說明**：Q&A 形式，記錄開發過程中的關鍵決策與解決方案

## 第一章：架構設計

**Q1：為什麼要從 v9.4.2 的單一腳本重構為 v10.0 的 MVC 架構？**

A：v9.4.2 的單一腳本雖然功能完整，但隨著功能增加，維護困難度指數成長。MVC 拆分能達到：
- **可測試性**：各服務模組可獨立單元測試
- **可維護性**：修改 DB 邏輯不影響評分引擎
- **可擴展性**：新增 AI 服務只需加一個 `Service_AI.gs`

---

**Q2：為什麼選擇 Google Sheets 作為資料庫，而非真正的資料庫？**

A：因為目標用戶（教師/教育機構）已有 Google Workspace 帳號，且 GAS ↔ Sheets 整合無需額外設定或費用。對於每次最多數十到百位學員的互動量，Sheets 效能完全足夠。

---

## 第二章：AI 整合

**Q3：Gemini API 呼叫時出現「models/gemini-3.0-flash not found」錯誤，如何處理？**

A：Gemini 3.0 尚未正式發布，API 不存在此模型。解決方案是回退至 `gemini-2.5-flash`（已確認可用的穩定版本）。
```javascript
// ❌ 錯誤
const model = 'gemini-3.0-flash';

// ✅ 正確
const model = 'gemini-2.5-flash';
```

---

**Q4：如何讓 Gemini AI 回傳結構化 JSON，而非自由文字？**

A：在 Prompt 中明確要求 JSON 格式，並指定欄位。範例 Prompt 結尾：
```
請以下列 JSON 格式回答，不要有任何額外文字：
{
  "feedback": "...",
  "suggestions": ["...", "..."],
  "encouragement": "...",
  "score_analysis": "..."
}
```
若 AI 回傳的字串包含 markdown 代碼塊（```json ... ```），需在程式中先清除再 `JSON.parse()`。

---

**Q5：什麼是「Coach Persona（教練人格）」？為什麼要導入？**

A：傳統 AI 回饋易流於「你的答案是對的/錯的」，缺乏激勵感。導入教練人格後，Prompt 加入指示：
- 以正向、鼓勵的語氣回饋
- 指出優點再提改進方向
- 避免直接說「錯了」，改為「試著想想…」

結果：學員實際互動體驗顯著提升。

---

## 第三章：Admin CMS

**Q6：為什麼課程階段列表要改為動態載入？**

A：原先 `UI_Admin.html` 的「新增題目」表單中，課程階段是硬編碼的靜態清單。若管理員透過試算表直接新增階段，CMS 畫面不會自動更新，造成資料不同步。

解決方案：新增 `apiGetStageList()` API，前端在表單開啟時呼叫此 API 取得最新列表。

---

**Q7：如何處理「新增題目時，若指定階段不存在」的情境？**

A：在 `apiAddQuestion()` 的處理邏輯中，先呼叫 `Service_DB.getStageById()` 確認階段存在。若不存在，自動呼叫 `Service_DB.createStage()` 建立新階段後，再寫入題目。前端只需在表單加一個「這是新階段」的勾選框。

---

## 第四章：安全與防護

**Q8：「防秒刷」機制是如何實作的？**

A：`Service_Security.gs` 中的 `withSubmitGuard_()` 函式：
1. 記錄每位學員最後一次提交的時間戳記至 Sheets
2. 每次提交時，比對當前時間與上次提交時間
3. 若間隔 < `SUBMIT_GUARD_SECONDS`（預設 10 秒），直接拒絕並回傳錯誤

---

## 第五章：部署常見問題

**Q9：部署後訪問 Web App 出現「You need permission」，如何解決？**

A：發布時「存取權限」必須設為「任何人（Anyone）」，而非「只有自己」。確認後重新部署即可。

---

**Q10：Script Properties 如何設定？**

A：在 GAS 編輯器：
1. 左側選單 → 「專案設定（齒輪圖示）」
2. 下滑至「指令碼屬性」→ 「新增屬性」
3. 屬性名稱：`GEMINI_API_KEY`，值：你的 API Key
4. 在程式中用 `PropertiesService.getScriptProperties().getProperty('GEMINI_API_KEY')` 讀取

## 第七章：自動化部署工具與環境排除 (clasp & Terminal)

**Q12：什麼是 clasp？為什麼要使用它？**

A：`clasp` (Command Line Apps Script Projects) 是 Google 官方提供的指令碼工具。
- **優點**：讓我們能在本地（如 VS Code）撰寫代碼，並像使用 Git 一樣透過指令同步到 Google 雲端。
- **核心價值**：避免手動複製貼上，支持本地開發環境的所有優點（如 AI 輔助、程式碼提示）。

---

**Q13：今日學習到的常用 clasp 指令有哪些？**

A：
1. `clasp login`：啟動 Google 帳號授權流程。
2. `clasp create --title "專案名稱" --type webapp`：在雲端建立新的指令碼專案。
3. `clasp push -f`：將本地所有檔案強制推送到雲端（會覆蓋雲端原有內容）。
4. `clasp open`：在瀏覽器中直接開啟當前專案的 GAS 編輯器網頁。

---

**Q14：執行 clasp 時出現「ReferenceError: require is not defined」是為什麼？**

A：這是因為本地的測試腳本（如 `local_test_runner.js`）使用了 Node.js 的語法（`require`），而這些檔案被同步到了不支援 Node.js 的 Google Apps Script 環境。
**解決方案**：建立 `.claspignore` 檔案，將所有非 `.gs` 或 `.html` 的本地檔案排除在同步範圍外。

---

**Q15：今日學習到的終端機 (Terminal) 基礎操作？**

A：
- **開啟終端機**：快捷鍵 `Ctrl + \`` 或選單「檢視 -> 終端機」。
- **路徑切換 (Windows)**：使用 `cd "路徑名稱"` 進入資料夾。若路徑包含空格或特殊符號（如「」），務必使用雙引號括起來。
- **分頁管理**：終端機視窗右側可切換不同進程（如編譯中、開發中、或是登入授權中）的分頁。

---

**Q16：如果 clasp 推送失敗，提示「User has not enabled the Apps Script API」，該怎麼辦？**

A：這代表您的 Google 帳號尚未開啟開發者權限。需前往 [script.google.com/home/usersettings](https://script.google.com/home/usersettings) 將「Google Apps Script API」切換為 **開啟 (On)**。

## 第八章：v10.1 同步與除錯馬拉松

**Q17：為什麼我修改了代碼，部署後網頁卻沒有變化？**

A：這是本次開發最核心的「同步陷阱」，原因有三：
1. **檔案字尾衝突 (.gs vs .js)**：雲端專案若已存在 `Main.js`，當你推送到 `Main.gs` 時，Google 會同時保留兩者。執行時可能優先抓舊的 `.js`，導致更新失效。
2. **瀏覽器/CDN 快取**：Google Web App 有嚴重的快取機制，必須透過「管理部署 -> 編輯 -> 新版本」強制生成新 URL 才能確保生效。
3. **試算表名稱誤導**：頂部標題若讀取自試算表名稱（v10.0），即使程式碼更新到 v10.1，視覺上看起來還是舊的。

**Q18：為什麼點擊登入按鈕後一直顯示「處理中...」卻沒反應？**

A：通常是伺服器端發生了「靜默崩潰（Silent Crash）」：
1. **ReferenceError**：例如在 v10.1.3 中，`result` 變數未定義即使用，導致 Google 伺服器直接中斷連線。
2. **Date 序列化失敗**：`google.script.run` **不能直接傳回 Date 物件**。必須先用 `.toISOString()` 轉為純文字，否則回傳流程會直接卡死。

**Q19：如何確保 `clasp` 推送是純淨的？**

A：
1. **核級清理法**：執行 `rm *.js` 確保本地沒有殘留舊檔案。
2. **強制覆蓋**：使用 `clasp push -f` 驅逐雲端的衝突檔案。
3. **檢查 .claspignore**：確保只有必要的 `.gs`、`.html` 與 `appsscript.json` 被上傳。

## 第九章：v10.3.0 多活動與同步深度排查

**Q20：為什麼要在 v10.3.0 導入多活動（Multi-Activity）架構？**

A：為了讓同一套系統能服務不同的班級或時段，而不需要重複部署多個 Web App。透過「動態路由」機制，管理員可以一鍵切換當前活躍的課程，實現資料的高效管理。

**Q21：多活動架構如何確保「向下相容」？**

A：
1. **預設機制**：當 `ACTIVE_ACTIVITY_ID` 為空時，系統自動導向舊有的 `系統設定` 與 `學員資料總表`。
2. **無縫銜接**：舊使用者不需要做任何搬移，系統邏輯會自動處理「有 ID」與「無 ID」的分支路徑。

**Q22：發生「clasp push 成功但 GAS 沒更新」的極端情況如何排查？**

A：這是今天最寶貴的除錯經驗：
1. **檢查 scriptId**：對比 `.clasp.json` 與 GAS 編輯器的網址列 ID，若不一致代表你推錯專案了。
2. **清除快取**：GAS 編輯器有 SPA 快取。
   - **無效**：F5。
   - **普通**：`Ctrl+Shift+R`。
   - **最有效**：**完全關閉該分頁並重新從網址進入**。

**Q23：為什麼更新了 `Main.gs` 裡的使用說明，Web App 介面卻沒變？**

A：因為 `apiGetManual` 是由 Web App 呼叫的。即使 `clasp push` 更新了原始碼，你必須去 GAS 介面進行「管理部署 -> 建立新版本」，Live 網頁才會讀取到最新的程式邏輯。

## 第十一章：v10.3.7 深度除錯紀錄（專案對齊與編碼修復）

**Q24：為什麼會發生「clasp push 成功，但編輯器沒更新」的情形？**

A：這通常是因為 `.clasp.json` 中的 `scriptId` 與您瀏覽器開啟的專案 ID 不一致。
*   **原因**：Google 帳號內可能存在多個同名或相似的專案（如舊版、備份版）。
*   **誤區**：只要 ID 是有效的且您有權限，`clasp` 就會成功推送。它並不知道您「肉眼正在看」的是哪一個。

**Q25：如何避免這種「平行時空」的同步錯誤？**

A：
1.  **核對網址列**：操作前，手動對比 `.clasp.json` 裡的 `scriptId` 是否與瀏覽器網址 `projects/[ID]/edit` 一致。
2.  **單一真相**：確保本地只有一個專案資料夾對應一個雲端專案，避免在不同裝置間混合使用舊的設定檔。

**Q26：為什麼之前更新版號時發生亂碼與語法錯誤？**

A：是因為使用 PowerShell 指令批次取代文字時，破壞了檔案的 **UTF-8 (無 BOM)** 編碼格式。
*   **影響**：導致中文註解變亂碼，甚至影響 JavaScript 執行效果。
*   **修復**：改用 Node.js 腳本（指定 `utf8` 讀寫）或 VS Code 的全域取代功能來確保編碼純淨。

**Q27：專案目錄結構在今日做了什麼重大調整？**

A：清理了根目錄下所有的 `.gs` 與 `.html` 檔案（由於歷史原因殘留的重複檔案），並在 `.clasp.json` 中明確指定 `"rootDir": "src"`。這確立了 **「src/ 為唯一開發真相」** 的原則，避免了 `clasp` 在推送時因檔案重複而報錯。

**Q28：為什麼會發生「clasp push 成功，但網頁沒更新」且最終發現是 Script ID 錯誤？**

A：這是一個極其隱蔽的陷阱。原因如下：
1. **多專案並存**：Google 帳號內可能存在多個名稱相似但 ID 不同的專案。
2. **clasp 盲目執行**：`clasp` 只會根據 `.clasp.json` 裡的 `scriptId` 進行推送。只要該 ID 對應的專案存在且你有權限，它就會回報「成功」。它並不知道這是不是你「真正想看」的那一個。
3. **對齊失效**：開發過程中若本地設定檔指向了舊的或備份用的專案 ID，推送就會失效。

**防範方法**：每次開始工作前，先將 `.clasp.json` 內的 `scriptId` 與瀏覽器網址列的 ID 進行對比。

**Q29：系統現在如何實現「完全免手動貼上網址」的完美自動化？**

A：在先前的版本中，系統直接抓取 `ScriptApp.getService().getUrl()`，但因為 Google 權限與執行環境的差異，選單觸發時會抓出一條限作者開發用的 `/dev` 網址，導致學員無法存取。
為了解決這個問題，同時保留「免手動貼上」的無縫體驗，目前的 v10.3.7 升級版實作了**自動補捉與轉換機制**：
1. 當您清空 M2/M3 並點開「開啟控制台」時。
2. 系統底層會抓到 `/dev` 測試網址。
3. 程式碼在背景瞬間**將字串 `/dev` 替換為 `/exec` 正式網址**。
4. 最終，系統會自動在 M2/M3 填入這組正確的正式網址，並顯示在控制台介面上。
這樣就達到了「免手動複製貼上」且「保證連結 100% 有效」的雙重體驗。
